## 1. The wire contract mobile must implement (Phase 0 — foundations)

All in `mobile/src/store/multiplayerStore.ts` + `mobile/src/services/websocket.ts`.

**[audit] Load-bearing precondition — spread-merge:** `duel`, `autoQueueInMs`, and `queueBoundDuo` ride **only** the initial snapshot (`Game.js:238/257/260`, `getInitialSendState`) and are absent from every per-round `getSendableState` broadcast (`Game.js:697-738`). Mobile's `...data` spread-merge into persisted `gameData` (`multiplayerStore.ts:735`) is what keeps them alive across round ticks. Never replace `gameData` wholesale on a `game` message; the merge is a wire-contract requirement, not a convenience.

### 1.1 Verify flag (do LAST — this is the rollout gate)
- `{type:'verify', ..., teamSupport: true}` — monotonic per session (`Player.js:113-115` **[verified]**), re-derived from the current socket on every reconnect (`Player.js:167`). Until sent, the server rejects mobile from: `createPrivateGame{mode:'2v2'}`, `joinPrivateGame`/`acceptInvite` into any team surface, `find2v2Match` (whole-lobby check → `teammateNeedsUpdate` toast), `setTeamConfig` enable (eviction), and reconnect into a team game (eject + toast).
- **The `duel:true` trap [verified]**: server sets `this.duel = duel || teamDuel` (`Game.js:42`) — every team2v2 game arrives with `duel:true` AND `team2v2:true`. Mobile's existing `gameData.duel` gates (`DuelHUD` mount, `showDuelMatchupIntro`, results `isDuel`) will mount 1v1 UI for 2v2 games the moment the flag flips. Every `duel` gate must become `duel && !team2v2` (1v1) or get a team branch — this is the single most dangerous porting footgun.
- **[audit] One `duel` gate that must KEEP firing for 2v2**: the round-1 anti-cheat warning banner (web `gameUI.js:981-991`, key `duelWarningText`, gated `duel && public && isStartingDuel`) fires for public team2v2 too. When auditing the duel gates, port this banner to the 2v2 intro and do not mis-gate it to 1v1-only. It was the one web surface (of 45 inventoried) with no line-item in the July 8 plan.
- **[audit] Old-build invite UX**: the `invite` payload (`ws.js:1042-1047`) carries no team indicator and the send path does no `teamSupport` check — a flagless friend receives the invite normally and only hits the rejection at `acceptInvite`. ### 11.1 What the web system is (the parity target)

- **SFX engine** (`audio.js:1-262`): lazy AudioContext; fetch+decode-once buffer cache with in-flight dedup; per-play options: mix `volume`, anti-repetition pitch jitter (±6% default), deterministic `rate` override (for pitch that carries MEANING — never jittered), per-sound 80ms cooldown with optional leading-edge `debounce`; `stopSfx(name, fade)` for long beds. **Prime directive, carries to mobile verbatim: every entry point swallows errors — audio must never break gameplay.**
- **Assets are MP3-only** (Safari ruling; also the safe RN codec). New-sound rule: transcode via ffmpeg/libmp3lame, drop in `public/sounds/`.
- **Volume model**: slider-space 0–1 persisted (`sfxVolume` default 0.85, `musicVolume` default **0.5 [raised from 0.45, July 10 PM]**); perceptual mapping **`toGain(v) = v²`** applied ONLY where a user-facing volume reaches an output gain (`audio.js:26-36`) — per-play mix ratios stay linear. Muted costs zero data: preloads skipped, music never streams.
- **[July 10 PM] Music resilience model — ports in SPIRIT, not mechanism** (web ref: the `startMusic` nudge funnel + wedge probe in `audio.js`). Three invariants the mobile engine must honor, and they matter MORE on mobile if music streams from prod over cellular (§11.4):
  1. A track that dies without a clean error must not leave the engine believing it's playing. Web detects the "starved corpse" (no error event, `paused` false, `currentTime` frozen) with a frozen-position probe across nudges; mobile should instead hook `expo-audio`'s player status events (error / stall / completion) — RN gets real callbacks where the web element goes silent.
  2. Every resume trigger funnels through ONE idempotent recovery entry point (web: `startMusic`, fed by clicks + the `'online'` event). Mobile's triggers: AppState foreground, NetInfo connectivity-restore, and player-status errors. Don't invent parallel resume paths.
  3. A failed play DISARMS and waits for the next trigger — never a hot retry loop while offline.
- **Music** (`audio.js:264-496`): streamed through an `<audio>` element, never decoded to buffers (a track costs memory only while playing); two playlists loudness-matched to ~-14.5 LUFS — `chill` (7 tracks) and `competitive` (5, `public/music/competitive/`); shuffle with no back-to-back repeat; fades: 2s in, **10s for the very first track of a session** (deliberate gentle first impression), 1s out, 1.5s playlist crossfade; hidden tab = immediate fade over 5s then pause (0s grace — user ruling); game-surface-only via `setMusicAllowed` (Home claims it on mount).
- **The competitive-playlist gate is `inMatchmadePipeline`** (`home.js:1440-1447`): `(inGame && gameData.public) || gameQueued || nextGameQueued`. **NEVER gate on `duel`** — unranked "duels" are public non-duel FFA server-side (same trap class as §1.1). The `nextGameQueued` term exists so Play Again doesn't blip chill between matches. Mobile translation: `inGame && gameData?.public`, any `gameQueued` value — which means the 2v2 stage-2 null-`gameData` window (§3) keeps competitive music for free via `gameQueued === '2v2'`.
- **Ducking**: `duckAudio(true/false)` collapses the master gain around interstitials (`home.js:2537-2538`; Poki QA requires silent audio during `commercialBreak`).
- **Preload discipline**: decode ahead of the moment that needs zero latency — `pin`/`guess` on map mount (`Map.js:358-362`), `ticking` when a timed round is possible (`gameUI.js:879-884`), `multinoti` whenever a live game could ping (`home.js:1449-1454`). ### 11.4 Music engine + native decisions
- **Engine: `expo-audio`** (mobile is SDK 54; `expo-av` is deprecated there — do NOT add it).
